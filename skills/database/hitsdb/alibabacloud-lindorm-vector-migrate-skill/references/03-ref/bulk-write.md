# Lindorm Data Write Interface

## Connection and Verification

```bash
# Search engine port is fixed at 30070
curl -s -u <username>:<password> "http://<host>:30070/"
```

## Create Index

### Check if Index Exists

```bash
curl -s -u <username>:<password> -o /dev/null -w "%{http_code}" \
  "http://<host>:30070/<index_name>"
# 200 = exists, 404 = does not exist
```

### Delete Existing Index (if clearing is needed)

```bash
curl -s -u <username>:<password> -X DELETE "http://<host>:30070/<index_name>"
```

### Create Index (PUT)

```bash
curl -s -u <username>:<password> -X PUT "http://<host>:30070/<index_name>" \
  -H "Content-Type: application/json" \
  -d '{
    "settings": {
      "index": {
        "knn": true,
        "number_of_shards": 3
      }
    },
    "mappings": {
      "_source": {"excludes": ["<vector_field>"]},
      "properties": {
        "<vector_field>": {
          "type": "knn_vector",
          "dimension": 128,
          "data_type": "float",
          "method": {
            "engine": "lvector",
            "name": "hnsw",
            "space_type": "cosinesimil",
            "parameters": {"m": 24, "ef_construction": 500}
          }
        },
        "text_field": {"type": "text", "analyzer": "ik_max_word"},
        "category": {"type": "keyword"},
        "price": {"type": "float"},
        "updated_at": {"type": "date", "format": "strict_date_optional_time"}
      }
    }
  }'
```

### DDL Construction Rules

1. **`settings.index.knn`**: Must be `true` to enable KNN search
2. **`settings.index.number_of_shards`** (optional): Primary shard count, **cannot be modified after creation**. Recommended 3~5 for over one million records
3. **`settings.index.knn_routing`** (optional): `true` enables custom routing key for multi-tenant isolation. Requires `number_of_shards > 1`
4. **`_source.excludes`**: Add vector field name to excludes to reduce storage overhead
5. **Vector field**: `type: knn_vector`, must specify `dimension`, `data_type`, `method`
6. **`method.engine`**: Fixed `lvector`
7. **`method.name`**: `hnsw` (online, queryable immediately after write) / `ivfpq` (offline build) / `ivfbq` (offline build)
8. **`method.space_type`**: `cosinesimil` / `l2` / `innerproduct`
9. **`method.parameters`**: Optional, e.g. `{"m": 24, "ef_construction": 500}` for HNSW
10. **Dynamic fields**: Not declared in DDL; Lindorm auto-infers via dynamic mapping

## Bulk Write (_bulk API)

```bash
curl -s -u <username>:<password> -X POST "http://<host>:30070/_bulk" \
  -H "Content-Type: application/x-ndjson" \
  -d '
{"index": {"_index": "<index_name>", "_id": "doc_001"}}
{"title": "Product A", "price": 99.9, "embedding": [0.1, 0.2, ...]}
{"index": {"_index": "<index_name>", "_id": "doc_002"}}
{"title": "Product B", "price": 59.9, "embedding": [0.3, 0.4, ...]}
'
```

### _bulk Format Description

- Each batch consists of alternating action lines + source lines; each line is a JSON object separated by newlines (NDJSON)
- The `_id` in the action line is derived from the source primary key field. **MUST use `str()` to convert to string** (Milvus INT64 primary keys, Qdrant int-type Point IDs, etc.); same `_id` overwrites (upsert semantics)
- `_source` does not include `_id`, `_sort`, or other internal fields
- Vector fields already excluded in `_source.excludes`; they can still be included during writes (they won't be stored)
- Routing write (optional): add `"routing": "value"` to the action line so documents with the same routing value land in the same shard

### Response Structure

```json
{
  "took": 30,
  "errors": false,
  "items": [
    {"index": {"_index": "<index>", "_id": "doc_001", "status": 201}},
    {"index": {"_index": "<index>", "_id": "doc_002", "status": 201}}
  ]
}
```

- `errors: true` indicates failed items; iterate `items` to find entries with `status >= 400`
- Status code 429 indicates rate limiting; use exponential backoff retry

## Parallel Writes and Backpressure Control

When `num_workers >= 2`, the migration script generated by the Agent uses `ThreadPoolExecutor` for parallel writes. **Must be paired with backpressure control** to prevent memory bloat when read speed far exceeds write speed.

### Core Pattern (Python, used by Agent when generating scripts)

> Recommended to use `elasticsearch-py`'s `helpers.bulk`, which is more concise and automatically handles connection reuse. Below is the primary example.

```python
from concurrent.futures import ThreadPoolExecutor, wait, FIRST_COMPLETED
from elasticsearch import Elasticsearch
from elasticsearch.helpers import bulk
import time, sys

NUM_WORKERS = 10
MAX_RETRIES = 3
BASE_DELAY = 1.0

es = Elasticsearch(
    hosts=[f"http://{host}:30070"],
    http_auth=(username, password) if username else None,
    request_timeout=60,
)

def write_batch(actions):
    """Single write task with rate-limit retry. actions is list[dict], each containing _index, _id, _source."""
    for attempt in range(MAX_RETRIES):
        try:
            success, errors = bulk(es, actions, raise_on_error=False)
            err_count = len(errors) if isinstance(errors, list) else 0
            return success, err_count
        except Exception as e:
            if "429" in str(e) or "rejected" in str(e).lower():
                time.sleep(BASE_DELAY * (2 ** attempt))
            else:
                raise
    return 0, len(actions)

def build_actions(batch, index_name, pk_field, routing_field=None):
    """Convert source data batch to elasticsearch-py bulk actions."""
    actions = []
    for row in batch:
        action = {
            "_index": index_name,
            "_id": str(row.get(pk_field, "")),
            "_source": {k: v for k, v in row.items() if k != "_sort"},
        }
        if routing_field and routing_field in row:
            action["_routing"] = str(row[routing_field])
        actions.append(action)
    return actions

executor = ThreadPoolExecutor(max_workers=NUM_WORKERS)
pending_futures = {}
total_migrated = 0
total_errors = 0
seq = 0

for batch in source_scan():
    actions = build_actions(batch, index_name, pk_field, routing_field)

    # Backpressure control: block when queue is full
    while len(pending_futures) >= NUM_WORKERS:
        done, _ = wait(pending_futures.values(), return_when=FIRST_COMPLETED)
        for done_future in done:
            done_seq = next(s for s, f in pending_futures.items() if f is done_future)
            del pending_futures[done_seq]
            ok, err = done_future.result()
            total_migrated += ok
            total_errors += err

    future = executor.submit(write_batch, actions)
    pending_futures[seq] = future
    seq += 1

# Drain remaining write tasks
while pending_futures:
    done, _ = wait(pending_futures.values(), return_when=FIRST_COMPLETED)
    for done_future in done:
        done_seq = next(s for s, f in pending_futures.items() if f is done_future)
        del pending_futures[done_seq]
        ok, err = done_future.result()
        total_migrated += ok
        total_errors += err

executor.shutdown(wait=False)
```

> **Alternative**: If `elasticsearch-py` cannot be installed, use `urllib.request` to directly construct NDJSON and call the `_bulk` endpoint (requires manual Basic Auth and NDJSON serialization).

### Backpressure Principle

```
Read speed >> Write speed (common in public network scenarios)

Without backpressure:          With backpressure:
┌──────┐                      ┌──────┐
│ Read  │──→ [100+ queued]    │ Read  │──→ [max N items]
└──────┘                      └──────┘
                                ↓ Block when queue full
                              ┌──────┐
                              │ Wait  │ until any write completes
                              └──────┘
```

- `NUM_WORKERS` controls both: max concurrent write threads **and** task queue limit
- Memory usage is bounded to `NUM_WORKERS × batch_size` documents

## max_errors Error Tolerance

`max_errors` controls the maximum number of write errors allowed during migration. `0` means stop on first error. After each batch write, accumulate error count; terminate with `[INTERRUPTED]` line when threshold is reached.

## IVFPQ/IVFBQ Offline Index Building

HNSW is queryable immediately after write. IVFPQ/IVFBQ require manually triggering a build after all data is written.

### Prerequisite: Enable Offline Build in DDL

When creating IVFPQ/IVFBQ indices, the DDL's `settings.index` **MUST** include `"knn.offline.construction": true`; otherwise the write phase attempts online building, resulting in extremely poor performance or errors. Example:

```json
{
  "settings": {
    "index": {
      "knn": true,
      "knn.offline.construction": true,
      "number_of_shards": 3
    }
  },
  "mappings": {
    "properties": {
      "<vector_field>": {
        "type": "knn_vector",
        "dimension": 128,
        "data_type": "float",
        "method": {
          "engine": "lvector",
          "name": "ivfpq",
          "space_type": "cosinesimil"
        }
      }
    }
  }
}
```

### Trigger Build

After all data is written, call the Build API:

```bash
curl -s -u <username>:<password> -X POST \
  "http://<host>:30070/_plugins/_vector/index/build" \
  -H "Content-Type: application/json" \
  -d '{"indexName": "<index_name>", "fieldName": "<vector_field>"}'
```

> **Common incorrect endpoint comparison** (Agent MUST NOT use these incorrect endpoints):
>
> | Incorrect approach | Why it's wrong | Correct approach |
> |-------------------|----------------|-----------------|
> | `POST /<index>/_forcemerge` | Standard ES API; only merges Lucene segments, **does not trigger** IVFPQ/IVFBQ vector index build | `POST /_plugins/_vector/index/build` |
> | `POST /<index>/_refresh` | Only refreshes search visibility; does not trigger build | `POST /_plugins/_vector/index/build` |
> | `PUT /<index>/_settings {"index.knn.offline.construction": false}` | Disabling offline build is not the same as triggering a build | `POST /_plugins/_vector/index/build` |

### Poll Build Status (MUST follow)

After triggering build, **MUST** poll until `status` is `ready` or `failed`; **MUST NOT** exit polling while `status=building`.

```bash
curl -s -u <username>:<password> -X GET \
  "http://<host>:30070/_plugins/_vector/index/tasks" \
  -H "Content-Type: application/json" \
  -d '{"indexName": "<index_name>", "fieldName": "<vector_field>", "taskIds": "[]"}'
```

Response: `{"payload": [{"status": "ready|building|failed", ...}, ...]}`

#### Polling Best Practices

Migration scripts generated by the Agent for IVFPQ/IVFBQ **MUST** automatically execute Build + polling after data write completes, following these rules:

| Rule | Requirement |
|------|-------------|
| **Polling interval** | First wait 30 seconds, then every 30 seconds thereafter. **MUST NOT** use 5-second or 10-second intervals; they trigger rate limiting (429) |
| **Rate limit backoff** | On HTTP 429, use exponential backoff: `delay = min(30 * 2^retry, 300)`, max 300 seconds |
| **Termination condition** | Exit polling **only** when `status=ready` or `status=failed`. **MUST** continue polling when `status=building` |
| **Timeout limit** | Max polling duration 2 hours (240 × 30 seconds); WARN on timeout but do not treat as failure |
| **HTTP 400/404** | Current index type does not need offline build (e.g. HNSW); exit normally |

```python
import time, requests

def build_and_poll(host, port, username, password, index_name, vector_field):
    base_url = f"http://{host}:{port}"
    auth = (username, password) if username else None

    # Trigger build
    resp = requests.post(
        f"{base_url}/_plugins/_vector/index/build",
        json={"indexName": index_name, "fieldName": vector_field},
        auth=auth
    )
    if resp.status_code in (400, 404):
        print(f"[INFO] Index type does not require offline build: {resp.text}")
        return "not_applicable"
    resp.raise_for_status()
    print(f"[INFO] IVFPQ/IVFBQ Build triggered, starting status polling (interval 30s)...")

    # Poll status
    max_polls = 240  # 2 hours
    poll_interval = 30
    for i in range(max_polls):
        time.sleep(poll_interval)
        try:
            resp = requests.get(
                f"{base_url}/_plugins/_vector/index/tasks",
                json={"indexName": index_name, "fieldName": vector_field, "taskIds": "[]"},
                auth=auth
            )
            if resp.status_code == 429:
                backoff = min(poll_interval * (2 ** (i % 4)), 300)
                print(f"[WARN] Rate limited, backing off {backoff}s before retry")
                time.sleep(backoff)
                continue
            resp.raise_for_status()
            tasks = resp.json().get("payload", [])
            if not tasks:
                print(f"[WARN] No build task returned, continuing to poll...")
                continue
            status = tasks[0].get("status", "unknown")
            print(f"[INFO] Build status: {status} ({i+1}/{max_polls})")
            if status == "ready":
                print("[INFO] IVFPQ/IVFBQ Build completed")
                return "ready"
            elif status == "failed":
                print(f"[ERROR] Build failed: {tasks[0]}")
                return "failed"
            # status == "building" → continue polling, MUST NOT exit
        except Exception as e:
            print(f"[WARN] Polling exception: {e}, continuing to retry...")

    print("[WARN] Build polling timed out (2 hours), please manually check build status")
    return "timeout"
```

- IVFPQ requires write row count > `nlist`; server minimum effective nlist=256
- KNN search fails or returns empty results when Build is not complete; **MUST** execute post-migration validation (Step 6) only after Build completes

## Key Notes

| Item | Description |
|------|-------------|
| Port | 30070 (fixed) |
| `_bulk` | Standard ES Bulk API (NDJSON format) |
| `_id` | Derived from `primary_field`; same `_id` overwrites (upsert) |
| `_source.excludes` | Vector fields excluded to save storage |
| `number_of_shards` | Primary shard count; cannot be modified after creation; recommended 3~5 for over one million records |
| `knn_routing` | Custom routing key; specify `routing` value during write to place same-value documents in the same shard |
| HNSW | Queryable immediately after write |
| IVFPQ/IVFBQ | Requires calling `_plugins/_vector/index/build` after write; poll `_plugins/_vector/index/tasks` until `ready`/`failed`. DDL MUST set `knn.offline.construction: true`. **MUST NOT** use `_forcemerge` as substitute |
