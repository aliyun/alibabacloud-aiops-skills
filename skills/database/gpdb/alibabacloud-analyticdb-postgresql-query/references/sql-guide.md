# SQL Guide

## Getting Table Structure from the Semantic Model

The Agent queries the `_agent_meta` schema to obtain available tables, columns, and JOIN relationships. **Using `\dt`, `\d`, or `information_schema` to discover table structure is prohibited.**

```bash
# Fetch the entire semantic model in one call (merged into 1 psql invocation)
psql \
  -c "SELECT t.table_name, t.description, t.domain,
            c.column_name, c.description AS col_desc, c.role, c.agg_function, c.synonyms
     FROM _agent_meta.tables t
     JOIN _agent_meta.columns c USING (table_schema, table_name)
     ORDER BY t.table_name, c.column_name;" \
  -c "SELECT source_table, source_column, target_table, target_column, join_type, description
     FROM _agent_meta.joins;"
```

The Agent learns from the results:
- Which tables are available and their business meanings
- The role of each column (dimension for grouping/filtering, measure for aggregation, identifier for primary/foreign keys)
- How to JOIN tables (only relationships declared in `_agent_meta.joins` may be used)
- Synonym mappings (when the user says "revenue", it maps to the amount column)

**Hard Constraint Rules**:
- SQL generated by the Agent may only reference tables registered in `_agent_meta.tables`
- Columns used by the Agent must be defined in `_agent_meta.columns`
- When a user request falls outside the semantic model scope, the Agent responds: "Current semantic model does not cover this data. Please contact DBA to extend `_agent_meta` definitions."

## Natural Language → SQL Conversion Rules

When the Agent receives a user's natural language query request:

1. Query `_agent_meta` to get the semantic model (available tables, columns, relationships)
2. Identify: target table, filter conditions, aggregation method, sorting, output fields (only selectable from within the semantic model scope)
3. Generate a standard PostgreSQL SELECT statement
4. Display the SQL for user confirmation before execution

**Notes**:
- For wide tables, select only necessary columns; avoid `SELECT *`
- For multi-table JOINs, prefer primary keys/distribution keys to reduce data shuffle
- All SELECT statements must include LIMIT (Gate 6 hard requirement)

## SELECT Syntax Reference

### Basic Query

```sql
SELECT col1, col2, agg(col3)
FROM table1 t1
[JOIN table2 t2 ON t1.key = t2.key]
WHERE conditions
GROUP BY col1, col2
HAVING agg_condition
ORDER BY col1 DESC
LIMIT n OFFSET m;
```

### JOIN

```sql
-- INNER JOIN
SELECT a.id, a.name, b.amount
FROM users a JOIN orders b ON a.id = b.user_id;

-- LEFT JOIN
SELECT a.name, COALESCE(SUM(b.amount), 0) AS total
FROM users a LEFT JOIN orders b ON a.id = b.user_id
GROUP BY a.name;
```

### Aggregation & Grouping

```sql
SELECT category, COUNT(*) AS cnt, SUM(value) AS total, AVG(value) AS avg_val
FROM items
GROUP BY category
ORDER BY total DESC;
```

### Window Functions

```sql
SELECT id, category, value,
    ROW_NUMBER() OVER (PARTITION BY category ORDER BY value DESC) AS rk,
    SUM(value) OVER (PARTITION BY category) AS category_total
FROM items;
```

### CTE (Subquery Organization)

```sql
WITH summary AS (
    SELECT user_id, SUM(amount) AS spend
    FROM orders
    WHERE created_at >= '2024-01-01'
    GROUP BY user_id
)
SELECT u.name, s.spend
FROM users u JOIN summary s ON u.id = s.user_id
ORDER BY s.spend DESC;
```

### Conditional Expressions

```sql
SELECT id,
    CASE WHEN status = 1 THEN 'Active' ELSE 'Inactive' END AS status_text,
    COALESCE(remark, 'N/A') AS remark
FROM items;
```

## Query Optimization Tips

- WHERE conditions on partition keys → partition pruning, reduced scan volume
- WHERE conditions include distribution keys → single-node execution, avoids broadcast
- JOIN conditions on distribution keys → local JOIN, avoids Redistribute Motion
- Prefer SQL-level aggregation to reduce exported data volume
- For large result sets, run `SELECT COUNT(*)` first to confirm volume before exporting
