# NOT Implementable via Public CLI/SDK — PAI Node

The following Node-related capabilities are **NOT exposed** through the public `aliyun paistudio` plugin or the public PAI OpenAPI (cross-check at https://next.api.aliyun.com/document/PaiStudio/2022-01-12/overview). They cannot be invoked from this skill. Cordon / Uncordon / Drain **are** in scope via `aliyun paistudio operate-node` (see `operate-node-parameters.md` + `confirmation-protocol.md`); everything else below is out of scope.

| Capability | Where it actually lives / Workaround |
| --- | --- |
| Node power (start / stop / reboot) | **NOT supported by this skill — refuse.** The agent MUST NOT execute or redirect to the underlying ECS instance's power-state CLI / SDK (whether kebab-case `aliyun ecs …` or legacy CamelCase actions), even when the user can name the ECS instance ID. Power-state mutation of a PAI-managed node is restricted to the platform **运维** team — escalation is **three-way only**: 平台**运维**团队 / Lingjun 运维工单 / 账号经理. **【refusal-text 禁字串】** The agent's refusal text MUST NOT contain the literal command names — see `refusal-patterns.md` *节点电源拒绝* row + `SKILL.md` §1 *节点 power / reboot 拒绝场景的禁字串规则*. Describe the forbidden path in natural language ("the underlying instance's power-state CLI") instead of echoing it. |
| `DrainParameters.Force` (force-drain a stuck Pod) | NOT exposed by this skill. Backend **运维** allowlist only. Agent MUST NOT include `"Force"` in `DrainParameters` and MUST NOT suggest `kubectl drain --force` / SSH `kill -9` as substitutes. Escalation three-way: 平台**运维**团队 / Lingjun 运维工单 / 账号经理. |
| `GetNode` / `GetNodeGPUMetrics` / `ListNodePods` (per-node metrics & pod listing) | NOT in public `aliyun paistudio` CLI. Direct user to **PAI 控制台** ResourceGroup / Quota aggregate dashboards. **Forbidden** as a fallback in the reply: `nvidia-smi` (whether via SSH, `kubectl exec`, or otherwise), `kubectl top node`, `kubectl describe node`, `kubectl get pods --field-selector spec.nodeName=...`, Prometheus / DCGM-exporter direct query, SSH-into-node. If node-level data is genuinely required, escalation three-way: 平台**运维**团队 / Lingjun 运维工单 / 账号经理. |
| Node replacement / migration | NOT exposed. Escalation three-way: 平台**运维**团队 / Lingjun 运维工单 / 账号经理. |
| Node label / taint mutation | NOT exposed. Cluster-internal control-plane operation; **forbidden** to suggest `kubectl label node` / `kubectl taint node` as a workaround. |
| Node attach / detach to a MachineGroup | Purchase-time only (`CreateMachineGroup` BSS flow). Not exposed via `paistudio`. |
| Cross-account node sharing | Not exposed publicly. |
| Real-time per-pod accounting on a node | Use `pai-quota-management` skill's `list-quota-workloads` instead. **Forbidden** to suggest `kubectl get pods --field-selector spec.nodeName=...` as a substitute. |

> **Rule**: if a capability is not in the public OpenAPI portal or in the `aliyun paistudio --help` output, treat it as **not implementable** by this skill. Do NOT fall back to SDK (Python / Java / Go) / `curl` / MCP / Terraform / `kubectl` (any subcommand) / SSH / `nvidia-smi` / any other 节点级直连. 节点属于平台托管资源，对用户和 Agent 而言不可直连——even when the user explicitly asks for one of these tools, the agent MUST refuse and MUST NOT echo back the command. If the user needs node mutation or node-level data, escalate via **平台运维团队 / Lingjun 运维工单 / 账号经理** (three-way) outside this skill.
