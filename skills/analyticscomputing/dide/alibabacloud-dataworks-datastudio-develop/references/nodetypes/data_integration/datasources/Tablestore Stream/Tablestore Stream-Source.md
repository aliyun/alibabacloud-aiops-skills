# Introduction

DataWorks Data Integration supports using Tablestore Stream Reader to read incremental data from Tablestore. This article introduces the data read capabilities of DataWorks for Tablestore Stream.

> **Supported Direction**: This data source only supports being used as a source (read), and does not support being used as a destination (write).

# Source

## Parameter Description

| Parameter | Description | Required | Default Value | Supported in Wizard Mode | Remarks |
|----------|----------|----------|--------|------------------|------|
| datasource | Data source name. Script mode supports adding data sources. The value of this configuration item must be consistent with the name of the added data source. | Y | None | Y |  |
| dataTable | The name of the table from which incremental data is exported. This table needs to have Stream enabled, which can be done when creating the table or by using the UpdateTable interface. | Y | None | Y |  |
| statusTable | The name of the table used by the Reader plugin to record status. This status can be used to reduce scanning of data outside the target range, thereby speeding up export. statusTable is a table used by the Reader to save state. If this table does not exist, the Reader will automatically create it. After an offline export task is completed, you do not need to delete this table. The status recorded in this table can be used for the next export task: * You do not need to create this table; just provide a table name. The Reader plugin will attempt to create this table under your instance. If the table does not exist, a new table is created. If the table already exists, it will check whether the table's Meta matches the expected Meta. If not, an exception will be thrown. * After an export is completed, you do not need to delete this table. The state of this table can be used for the next export task. * This table will have TTL enabled, and data will automatically expire. The data volume is considered to be very small. * For Reader configurations of multiple different dataTables under the same instance, the same statusTable can be used. The recorded status information does not interfere with each other. You can configure a name similar to TableStoreStreamReaderStatusTable. Please note not to use the same name as business-related tables. | Y | None | Y |  |
| startTimestampMillis | The left boundary of the incremental data time range (left-closed, right-open), in milliseconds: * The Reader plugin will find the position corresponding to startTimestampMillis from the statusTable and start reading and exporting data from that position. * If the corresponding position cannot be found in the statusTable, it will start reading from the first piece of retained incremental data and skip data with a write time earlier than startTimestampMillis. | N | None | Y |  |
| endTimestampMillis | The right boundary of the incremental data time range (left-closed, right-open), in milliseconds: * After the Reader plugin starts exporting data from the startTimestampMillis position, it ends the data export when it encounters the first data record with a timestamp greater than or equal to endTimestampMillis. The export is then complete. * When all current incremental data has been read, the reading will end even if endTimestampMillis has not been reached. | N | None | Y |  |
| date | Date format is yyyyMMdd, for example 20151111, indicating to export data for that day. If date is not specified, you need to specify startTimestampMillis and endTimestampMillis or startTimeString and endTimeString, and vice versa. For example, Caiyunjian scheduling only supports day-level granularity, so this configuration is provided. Its function is similar to startTimestampMillis and endTimestampMillis or startTimeString and endTimeString. | N | None | Y |  |
| isExportSequenceInfo | Whether to export time sequence information. Time sequence information includes data write time, etc. The default value is false, meaning it is not exported. | N | false | Y |  |
| maxRetries | The maximum number of retries per request when reading incremental data from Tablestore. The default is 30 retries. There are intervals between retries. The total time for 30 retries is approximately 5 minutes. Typically, there is no need to change this. | N | 30 | Y |  |
| startTimeString | The start time of the task, i.e., the left boundary of the incremental data time range (left-closed, right-open), in the format `yyyymmddhh24miss`, in seconds. | N | None | Y |  |
| endTimeString | The end time of the task, i.e., the right boundary of the incremental data time range (left-closed, right-open), in the format `yyyymmddhh24miss`, in seconds. | N | None | Y |  |
| enableSeekIterator | The Reader plugin needs to determine the incremental position first before pulling data. For frequently run tasks, the plugin will determine the position based on previously scanned positions. If the plugin has not been run before, it will start scanning from the beginning of the incremental data (by default, incremental data is retained for 7 days, i.e., 7 days ago). Therefore, when the scanning has not yet reached data after the set start time, there may be a period at the beginning with no data export. You can add `"enableSeekIterator": true` in the reader's configuration parameters to help speed up position determination. | N | false | Y |  |
| mode | Export mode. When set to single_version_and_update_only, it is row mode. If not set, the default is column mode. | N | None | Y |  |
| isTimeseriesTable | Whether it is a time series table. This configuration only takes effect in row mode, i.e., when mode is single_version_and_update_only. | N | false | Y |  |
| column | Column configuration. In **single_version_and_update_only** mode, the data columns to export. Configuration example: ```json "column":[ {"name":"pk1"}, {"name":"col1"}, {"name":"col2","dataType":"new"}, {"name":"col2","dataType":"old"}, {"name":"col2","dataType":"latest"} ], ``` * The `name` field specifies the name of the data column to export and must be configured. * The `dataType` field specifies the data type to export. The default is `new` type and is optional. `dataType` supports three enum types: * `new`: The value after the column is updated * `old`: The value before the column is updated * `latest`: The current latest value of the column in the table **Note** In row mode, this **must be configured**, otherwise no data will be exported. | * Row mode: Yes * Column mode: No | None | Y |  |

> Y = Yes, N = No
