# CLI Command Reference — Smart Voice Notification

## 1. SubmitIntent — Initiate a Smart Voice Call

| Attribute | Value |
|---|---|
| Action | `SubmitIntent` |
| CLI subcommand | `submit-intent` |
| API version | 2017-05-25 |
| Status | Intranet-online build (requires the intranet-online CLI plugin) |
| HTTP method | POST |
| Protocol | HTTPS / RPC |

### Plugin Installation (required)

> **Note**: `SubmitIntent` is currently distributed via the intranet-online CLI plugin only; the public-release plugin has not yet been synchronized with this command.

```bash
# Note: `aliyun plugin install` is a CLI tooling command and DOES NOT accept `--user-agent`.
aliyun plugin install --names dyvmsapi \
  --version 0.1.0 \
  --source-base https://cli.aliyun-inc.com/registry_id/2/env/online/plugins
```

### Business Parameters

| Parameter | Type | Required | Description |
|---|---|---|---|
| `--user-message` | string | **Yes** | Natural-language description of the call intent. The system matches the most relevant contact in the address book and the LLM generates the spoken content before placing the call. |

> Aside from `--user-message`, the other parameters listed by `--help` (`--extend-function`, `--lowcode-logic-app-name`, `--lowcode-physic-app-name`, `--lowcode-tenant-id`, `--route-name`) are internal low-code/routing pass-through fields and **do not need to be supplied by external users**.

### Common Global Parameters

| Parameter | Type | Description |
|---|---|---|
| `--region` | string | Override the default region (dyvmsapi is centralized; usually no need to specify) |
| `--cli-ai-mode` | bool | Enable AI mode for this single invocation |
| `--endpoint` | string | Override the service endpoint |
| `-q, --quiet` | bool | Quiet mode; suppress output |

### Invocation Example

```bash
# Basic invocation
aliyun dyvmsapi submit-intent \
  --user-message "Remind Dad to come home early" \
  --user-agent "AlibabaCloud-Agent-Skills/alibabacloud-vms-smart-call-by-tts/1.0.0"
```

### Successful Response Example

```json
{
  "Code": "OK",
  "Model": {
    "CallContent": "爸，早点回家啊。",
    "CallId": "1779785134472^9876543210",
    "Tag": "爸爸"
  },
  "RequestId": "56A1AFDB-6945-505A-938A-9B311D64B5C0",
  "Success": true
}
```

> **For phone-number safety, the response does not return the dialed number (PhoneNumber)**.

### Response Field Reference

| Field | Type | Description |
|---|---|---|
| `Code` | string | `OK` indicates the request was accepted |
| `Success` | bool | Whether the call succeeded |
| `Model.CallContent` | string | The actual playback text generated by the LLM |
| `Model.CallId` | string | Call ID (formatted as `<ts>^<ts>`); used for follow-up tracking |
| `Model.Tag` | string | The matched address-book tag |
| `RequestId` | string | Unique request identifier |

### Error Codes

| Code | Meaning | Suggested Handling |
|---|---|---|
| `InvalidParameter` | Parameter validation failed (UserMessage is empty, too long, etc.) | Correct UserMessage and retry |
| `ContactBookEmpty` | The current user's address book is empty | Add contacts in the console first, then retry |
| `AiCalleeNotFound` | No contact in the address book matched the intent's semantics | Verify the tag referenced in UserMessage matches one the user has registered; relay the error to the user |
| `ContentRefused` | The call intent was blocked by content-safety rules | Modify UserMessage and remove disallowed content |
| `UserNotInWhitelist` | Account is not yet on the SubmitIntent whitelist | Apply for access; not self-serve |
| `MissingSecurityToken` | Using STS temporary credentials but `SecurityToken` was not provided | Do **not** manually `export ALIBABA_CLOUD_SECURITY_TOKEN` in the shell — cross-process env cannot be propagated to subprocesses; this approach is technically infeasible. On the script path, invoke through the helper: `python3 scripts/run_with_aliyun_creds.py -- python3 scripts/dyvmsapi_rpc.py ...`; the helper auto-extracts `sts_token` from the `aliyun configure` StsToken profile and injects it into env. On the CLI path, confirm the current profile mode is `StsToken` and `sts_token` is not expired. **Strictly forbidden** to pass `SecurityToken` as a business parameter via `-P SecurityToken=...` — it is a public parameter that must participate in signing |
| `SystemError` | Internal system error | Retry once; if it still fails, contact technical support |
| `Throttling` | Per-number rate limit (1/min, 5/hour, 20/24h per number) | Wait 60 seconds, then retry |

## 2. QueryCallDetailByCallId — Query Call Detail Record

| Attribute | Value |
|---|---|
| Action | `QueryCallDetailByCallId` |
| CLI subcommand | `query-call-detail-by-call-id` |
| API version | 2017-05-25 |
| Status | Available in **both** the intranet-online plugin (`--source-base https://cli.aliyun-inc.com/...`, primary path) and the public-release plugin (`aliyun plugin install --names dyvmsapi`, external-fallback path). The CLI subcommand and parameters are identical across both builds. |
| HTTP method | POST |
| Protocol | HTTPS / RPC |

### Use Case

A successful `SubmitIntent` only means the call has been **accepted**. To confirm whether the callee answered, how long the call lasted, and when it ended, query this API.

### Business Parameters

| Parameter | Type | Required | Description |
|---|---|---|---|
| `--call-id` | string | **Yes** | The `Model.CallId` from `SubmitIntent`'s response, e.g. `"1779785134472^9876543210"` |
| `--prod-id` | int | **Yes** | **Must be `11000000300006`** (the Voice Notification product ID). `SubmitIntent` is dispatched under this product; any other value yields `Code: OK` with no `Data`. |
| `--query-date` | int | **Yes** | Unix-millisecond timestamp on the same day as `CallId`. Recommended: take the front segment of `CallId`: `${CALL_ID%%^*}`. |

### Invocation Example

```bash
CALL_ID="1779785134472^9876543210"
QUERY_DATE="${CALL_ID%%^*}"

aliyun dyvmsapi query-call-detail-by-call-id \
  --call-id "$CALL_ID" \
  --prod-id 11000000300006 \
  --query-date "$QUERY_DATE" \
  --user-agent "AlibabaCloud-Agent-Skills/alibabacloud-vms-smart-call-by-tts/1.0.0"
```

### Successful Response Example

```json
{
  "Code": "OK",
  "Message": "OK",
  "RequestId": "5A17BFE8-D155-5771-BF27-D29099EB997A",
  "Data": "{\"callId\":\"1779786101083^1779786101083\",\"endDate\":\"2026-05-26 17:01:51\",\"stateDesc\":\"用户接听\",\"callee\":\"185****8980\",\"bStartTime\":\"2026-05-26 17:01:47\",\"gmtCreate\":\"2026-05-26 17:01:41\",\"duration\":4,\"calleeShowNumber\":\"0571000018363\",\"bRingTime\":\"2026-05-26 17:01:42\",\"bEndTime\":\"2026-05-26 17:01:51\",\"state\":\"200000\",\"startDate\":\"2026-05-26 17:01:47\",\"hangupDirection\":0}"
}
```

> **Note 1**: `Data` is a JSON **string**; a second `JSON.parse` is required to access its fields.
> **Note 2**: in the example above, `callee` is masked as `185****8980`; **the API actually returns the full number** (the account owner viewing their own CDR — this does not contradict the SubmitIntent rule that phone numbers are not passed to the LLM).

### `Data` Inner-Field Reference

| Field | Type | Description |
|---|---|---|
| `callId` | string | Same as the request `CallId` |
| `state` | string | State code: `200000` = answered / others = not answered or anomaly |
| `stateDesc` | string | Localized state description: `用户接听` (answered) / `未接听` (not answered) / `呼叫中` (in progress), etc. |
| `duration` | int | Call duration in seconds (from answer to hangup) |
| `gmtCreate` | string | Time the request was accepted |
| `bRingTime` | string | Ring start time |
| `bStartTime` / `startDate` | string | Answer time |
| `bEndTime` / `endDate` | string | Hangup time |
| `callee` | string | Dialed number (**this API returns the full number**) |
| `calleeShowNumber` | string | Caller display number |
| `hangupDirection` | int | Hangup direction; `0` = callee hung up |

### Common Scenarios

| Symptom | Explanation |
|---|---|
| `Code: OK` is returned but **`Data` field is missing** | Means the record cannot be found yet: 1) scheduling delay (CDR usually queryable ~40s after the call ends; if still missing, retry after another 30–60s); 2) wrong prod-id (must be `11000000300006`); 3) query-date not on the same day as CallId |
| `Data` is present but `state ≠ 200000` | Not answered or anomalous hangup; relay `stateDesc` to the user |
| `state = 200000` and `duration > 0` | Answered with a non-zero talk duration |

### Recommended Presentation Format (readable)

Take the example above. After parsing, render the result as **three Markdown tables** to the user. **Do not** dump raw JSON, and **do not** use ASCII separators (`─────`, `===`, etc.) — those get absorbed as `<hr>` by Markdown renderers and reflow the rest of the text into a single line:

**Call Result**

| | |
|---|---|
| ✅ Status | `用户接听` (state=`200000`) |
| 👤 Callee | `185****8980` |
| 📲 Display No. | `0571000018363` |
| ⏰ Duration | `4` seconds |

**Timeline**

| | |
|---|---|
| 🕐 Accepted | `2026-05-26 17:01:41` |
| 🔔 Ringing | `2026-05-26 17:01:42` (+`1`s) |
| 📞 Answered | `2026-05-26 17:01:47` (+`5`s after ring) |
| 🔚 Hung up | `2026-05-26 17:01:51` (talk `4`s) |
| 👋 Hangup by | `callee-initiated` (hangupDirection=`0`) |

**Reference**

| | |
|---|---|
| 🆔 CallId | `1779786101083^1779786101083` |

**Masking rule**: replace the middle 4 digits of `callee` with `****` (e.g. `18546048980` → `185****8980`) so the full number never enters the chat context.

**When the call did not connect** (`state ≠ 200000`), omit the **Timeline** answered/hung-up rows and use a one-line summary, e.g.:

> ✗ Callee did not answer (`未接听`, state=`<actual not-answered code>`); rang for 25 seconds, then hung up.

(The state code above is illustrative — the actual code returned by the API is authoritative.)
